§1  PURPOSE AND SCOPE
================================================================================

This document replaces the older technical analysis and describes the code as
it exists now. The main objective is accuracy with respect to the current
implementation, not historical intent.

The analysis covers:
  - How game state is represented internally
  - How heuristic scoring works
  - How Monte Carlo Tree Search is implemented
  - Which improvements are available and which are active by default
  - How local simulation and tournament tooling are wired

This analysis does NOT present tournament outcomes, ablation tables, ELO
rankings, or any new result claims. Those may exist elsewhere in the
repository, but they are outside the scope of this updated write-up.

================================================================================
§2  CURRENT AGENT VARIANTS AND ENTRYPOINTS
================================================================================

The live entrypoint is main.py:

  AGENT_MODE = os.environ.get("SNAKE_TACTIC", "mcts").strip().lower()

So the default production path is the mode string "mcts".

agent_server.py maps mode strings to behaviours:

  random
    Uniform random legal move.

  heuristic
    Pure heuristic policy based on heuristic_move().

  mcts
  mcts_all
    Use ALL_IMPROVEMENTS_CONFIG.

  mcts_vanilla
    No priors, no opponent-aware root model, no heuristic rollouts,
    no RAVE, no UCB1-Tuned.

  mcts_heuristic_rollout
    Adds only heuristic rollouts.

  mcts_prior
    Adds priors and PUCT-style prior bonus, but no opponent-aware root model.

  mcts_opponent
    Adds opponent-aware root priors on top of priors and heuristic rollouts.

  mcts_rave
    Enables RAVE in addition to priors, opponent-aware root scoring,
    and heuristic rollouts.

  mcts_ucb1_tuned
    Enables UCB1-Tuned instead of standard UCB1 exploration.

Important current detail:
  The default contest path is "mcts", which resolves to ALL_IMPROVEMENTS_CONFIG.
  In the current code, ALL_IMPROVEMENTS_CONFIG enables:
    - prior-guided expansion
    - PUCT-style prior bonus
    - opponent-aware root scoring
    - heuristic rollouts
    - UCB1-Tuned
  but NOT RAVE.

This matters because older documentation described "all improvements" as if
RAVE were active simultaneously. That is not true in the current code.

================================================================================
§3  INTERNAL STATE REPRESENTATION
================================================================================

agent_core.py converts the Battlesnake API JSON into a compact internal dict:

  state = {
      "turn": int,
      "width": int,
      "height": int,
      "food": set[(x, y)],
      "hazards": {(x, y): stack_count},
      "snakes": [
          {
              "id": str,
              "name": str,
              "health": int,
              "body": [(x, y), ...],   # head first
          },
          ...
      ],
      "you_id": str,
  }

This representation is chosen for search speed:
  - Food is a set for O(1) membership checks.
  - Hazards are aggregated into a dictionary keyed by coordinate.
  - Snake bodies are stored as tuples so they can participate in sets.

make_state_from_game() performs the conversion once per top-level decision.
MCTS and heuristic code then operate on this internal structure only.

The most important helper tied to representation is moving_body_cells(state).
It returns the cells that remain occupied for the next step, excluding a tail
that will move away unless the snake has just grown. This is critical because
tail-motion handling determines whether a move is truly legal.

================================================================================
§4  CORE MOVEMENT AND SIMULATION LOGIC
================================================================================

The codebase centralises movement semantics in agent_core.py rather than
duplicating rules inside the heuristic and MCTS modules.

Important helpers:

  add_move(pos, move_name)
    Applies one of {"up","down","left","right"} to a coordinate.

  in_bounds(pos, width, height)
    Board boundary test.

  get_snake(state, snake_id)
    Fetches a snake dict from state.

  get_legal_moves(state, snake_id)
    Generates legal moves by:
      1. Building occupied cells with moving_body_cells()
      2. Removing the controlled snake's own tail
      3. Rejecting out-of-bounds and occupied destinations

  advance_state(state, planned_moves=None, stochastic=False)
    Applies one full simultaneous turn.

advance_state() is the transition function used by both heuristic lookahead and
MCTS. Its behaviour is important:

  1. Every living snake receives a move.
     - The controlled snake may receive a forced move via planned_moves.
     - Unspecified snakes use choose_greedy_move().

  2. New heads are inserted simultaneously.

  3. Food handling is resolved.
     - If a snake enters food, its health becomes 100 and its tail does not pop.
     - Otherwise health decreases by 1 and the tail advances.

  4. Hazard damage is applied after the move.
     - Damage is 14 × stack_count for the head cell.

  5. Deaths are resolved.
     - Out-of-bounds and health <= 0
     - Head-to-head collisions by length
     - Head into any body segment

  6. Dead snakes are removed, eaten food is removed, and turn increments.

This makes advance_state() the single most important shared rules engine in the
repository.

================================================================================
§5  HEURISTIC SYSTEM
================================================================================

The heuristic policy is layered rather than single-stage:
  - local_move_score() evaluates immediate move safety and tactical value.
  - evaluate_state() evaluates a resulting position more globally.
  - heuristic_move_scores() combines the two.
  - rollout_move() is a lighter-weight variant used during MCTS rollouts.

------------------------------------------------------------------------------
§5.1  Legal move generation and fallback behaviour
------------------------------------------------------------------------------

If no legal moves are available, fallback_move() returns the first in-bounds
direction in MOVE_ORDER, or "up" as a last resort.

This fallback is intentionally simple. It is not a strategic policy; it is a
safety net for malformed or terminal states so the agent always returns a move.

------------------------------------------------------------------------------
§5.2  local_move_score
------------------------------------------------------------------------------

local_move_score(state, snake_id, move_name) is the first tactical filter.
It returns negative infinity for immediately fatal actions and a numeric score
for legal actions.

Hard invalidation:
  - Destination out of bounds
  - Destination inside occupied cells after tail adjustment
  - Post-move health <= 0 after hunger and hazard damage
  - Zero reachable area after the move

Scoring terms in the current implementation:

  1. Reachable space
     score += reachable * 4.0
     reachable is computed by flood_fill() from the destination after marking
     the old head as occupied. This is the main anti-trap signal.

  2. Too-small pocket penalty
     if reachable < snake_length:
       score -= 120 + 18 * shortfall
     This heavily penalises entering a region that cannot physically contain
     the full snake body.

  3. Enemy head zones
     If an enemy of length >= ours can also reach the destination in one move:
       score -= 150
     If the enemy is shorter:
       score += 20
     enemy_head_zones() precomputes one-step opponent head reachability.

  4. Open neighbours
     score += 8 * open_neighbors
     This prefers squares with multiple exits over narrow corridors.

  5. Hazard penalty
     score -= 35 * hazard_stacks
     score -= max(0, 45 - post_health)
     Hazards are discouraged, especially when health becomes low.

  6. Direct food bonus
     If moving onto food:
       health < 35  -> +120
       health < 60  -> +70
       else         -> +25

  7. Food distance penalty
     food_dist = nearest_food_distance(state, snake_id)
     urgency is:
       3.5 if health < 25
       1.8 if health < 50
       0.5 otherwise
     score -= food_dist * urgency

  8. Centre bias
     score -= Manhattan distance from board centre
     This is a weak positional tie-breaker toward central mobility.

  9. Enemy proximity
     Within Manhattan distance <= 2:
       - larger/equal enemies create a penalty
       - smaller enemies create a smaller positive chase incentive

Operationally, local_move_score() answers:
  "If I take this move right now, does it keep me alive and tactically sane?"

------------------------------------------------------------------------------
§5.3  evaluate_state
------------------------------------------------------------------------------

evaluate_state(state, snake_id) is the heavier positional evaluator used by
the heuristic policy for one-step lookahead.

Terminal shortcuts:
  - Dead: return -1_000_000.0
  - Sole survivor: return 1_000_000.0 + length * 100 + health

Non-terminal scoring terms:

  1. Constant living offset
     +260.0

  2. Length
     +35.0 per body segment

  3. Health
     +1.2 per HP

  4. Flood-fill area
     +6.0 per reachable cell

  5. Safe-move count
     +22.0 per legal move

  6. Territory
     +4.0 per territory_score unit

territory_score() is a Voronoi-like estimate:
  - Compute our BFS distance map
  - Compute one BFS map per enemy
  - A cell scores 1.0 if we reach it strictly earlier than all enemies
  - A tied cell scores 0.2
  - Search depth is capped at 24

Additional penalties and bonuses:
  - Food distance is penalised with stronger urgency than in local_move_score
  - Hazard occupancy penalises the current head cell
  - Larger nearby enemies penalise the state
  - Smaller nearby enemies create a positive pressure to attack

evaluate_state() is therefore more strategic than local_move_score():
  it values space control, future mobility, and territorial dominance.

------------------------------------------------------------------------------
§5.4  mcts_evaluate_state
------------------------------------------------------------------------------

mcts_evaluate_state(state, snake_id) is a separate evaluator specifically tuned
for MCTS rollouts. This is an important current-code distinction.

Unlike evaluate_state(), it is normalised and clamped to [-1.0, 1.0].

Terminal values:
  - Dead -> -1.0
  - Sole survivor -> +1.0

Major design differences from evaluate_state():

  1. Relative length matters more explicitly
     The score includes both:
       - absolute length bonus
       - pairwise length-difference terms against each opponent
     This makes MCTS reward growth and matchup advantage, not just survival.

  2. Board features are scaled as fractions
     Reachable area and territory are divided by board_area.
     This keeps the reward bounded and stable for tree search.

  3. Food remains important even when health is high
     Distance to food is always penalised, with stronger urgency at low health
     but still a non-zero incentive when healthy.

  4. Health and hazard terms are small but continuous
     The reward stays smooth rather than dominated by large raw constants.

This function is what the MCTS tree ultimately tries to maximise.
The heuristic and MCTS policies are therefore related, but not identical.

------------------------------------------------------------------------------
§5.5  heuristic_move_scores and heuristic_move
------------------------------------------------------------------------------

heuristic_move_scores(state, snake_id) combines immediate and one-step future
evaluation:

  total(move) =
      local_move_score(state, snake_id, move)
      + 0.85 * evaluate_state(next_state, snake_id)

where next_state is generated by:

  advance_state(state, planned_moves={snake_id: move}, stochastic=False)

This means opponents are not frozen. Unspecified opponents are advanced using
their own greedy choices through choose_greedy_move().

Additional corrections:
  - If our snake dies in next_state, total = -inf
  - If next_state leaves us with zero legal moves, total -= 180
  - If the number of living snakes decreases, total += 50

choose_heuristic_move() then selects the argmax over these totals.
heuristic_move(game_state) is the API-level wrapper used by the heuristic mode.

------------------------------------------------------------------------------
§5.6  rollout_move
------------------------------------------------------------------------------

rollout_move(state, snake_id) is the light rollout policy used by MCTS when
heuristic rollouts are enabled.

It does NOT call evaluate_state().
Instead it:
  1. Scores legal moves with local_move_score()
  2. Sorts them
  3. Builds a shortlist of moves within 18 points of the best
  4. Chooses randomly from that shortlist

This is a compromise between two extremes:
  - purely random rollouts, which are cheap but noisy
  - fully deterministic greedy rollouts, which are informative but can become
    overly biased and reduce search diversity

================================================================================
§6  MCTS ARCHITECTURE
================================================================================

The MCTS implementation lives in MCTS.py and is controlled by the MCTSConfig
dataclass. The live wrapper is mcts_move().

------------------------------------------------------------------------------
§6.1  Config structure and active presets
------------------------------------------------------------------------------

MCTSConfig fields:

  use_priors: bool = True
  prior_bonus_scale: float = 4.0
  puct_decay: bool = True

  use_opponent_model: bool = True
  minimax_top_n: int = 2

  use_heuristic_rollout: bool = True
  rollout_depth: int = 8

  use_rave: bool = False
  rave_k: float = 500.0

  use_ucb1_tuned: bool = False
  exploration: float = 1.05

Preset summary:

  VANILLA_CONFIG
    All optional improvements disabled.

  HEURISTIC_ROLLOUT_CONFIG
    Rollouts guided, but no priors and no opponent-aware root priors.

  PRIOR_GUIDED_CONFIG
    Priors + heuristic rollouts.

  OPPONENT_AWARE_CONFIG
    Priors + opponent-aware root priors + heuristic rollouts.

  RAVE_CONFIG
    Opponent-aware priors + heuristic rollouts + RAVE.

  UCB1_TUNED_CONFIG
    Opponent-aware priors + heuristic rollouts + UCB1-Tuned.

  ALL_IMPROVEMENTS_CONFIG
    Priors + opponent-aware priors + heuristic rollouts + UCB1-Tuned
    with use_rave=False.

Current default:
  "mcts" -> ALL_IMPROVEMENTS_CONFIG

Time budget:
  If max_iterations is not supplied, mcts_move() reads MCTS_BUDGET from the
  environment and defaults to 0.92 seconds.

------------------------------------------------------------------------------
§6.2  Node structure
------------------------------------------------------------------------------

Each Node stores:
  - state
  - snake_id
  - parent
  - move that led to this node
  - children
  - priors per move
  - untried_moves ordered by descending prior
  - visits
  - value
  - value_sq
  - rave_visits
  - rave_value

value_sq exists specifically for UCB1-Tuned, because that selection formula
requires an empirical variance estimate.

For non-root children, priors are intentionally cheap:
  the code does not call heuristic_move_scores() for every child node.
  Instead it assigns 0.0 priors to legal moves. This is an explicit
  performance trade-off to preserve iteration count.

------------------------------------------------------------------------------
§6.3  Selection
------------------------------------------------------------------------------

tree_policy() walks down the tree until it finds:
  - a non-fully-expanded node, or
  - a terminal node

When a node is fully expanded, selection uses Node.best_child().

Base exploit term:
  exploit = child.value / child.visits

Exploration:
  Standard UCB:
    exploration * sqrt(log(parent_visits) / child_visits)

UCB1-Tuned alternative:
  variance = value_sq / n - exploit^2
  tuned = variance + sqrt(2 * log(parent_visits) / n)
  explore = exploration * sqrt(log(parent_visits) / n * min(0.25, tuned))

Prior bonus:
  If use_priors is enabled, the parent's prior scores are min-max normalised
  over siblings and added as:

    norm_prior * prior_bonus_scale / (1 + n)

if puct_decay is True.

This is PUCT-like behaviour rather than textbook UCT. The heuristic prior has
its strongest effect early and decays as empirical visit counts rise.

RAVE blend:
  If use_rave is enabled and RAVE stats exist for the move, exploit becomes a
  blend between empirical exploit and AMAF exploit using:

    beta = sqrt(rave_k / (3 * n + rave_k))

In the current default configuration, this branch is disabled.

------------------------------------------------------------------------------
§6.4  Expansion
------------------------------------------------------------------------------

expand() pops the first move from untried_moves, which are already ordered by
descending prior score.

The next state is produced by:

  advance_state(
      self.state,
      planned_moves={self.snake_id: move_name},
      stochastic=True,
  )

The stochastic flag matters because unplanned snakes then choose moves via
choose_greedy_move(..., stochastic=True), which introduces randomness among
near-top heuristic actions. This increases variation during search.

------------------------------------------------------------------------------
§6.5  Opponent-aware root priors
------------------------------------------------------------------------------

The root can receive more expensive priors than internal nodes.

likely_moves_for_snake(state, snake_id, top_n=2):
  - score the opponent's legal moves with heuristic_move_scores()
  - keep up to two plausible moves, with score-window filtering

opponent_joint_responses(state, snake_id, max_joint=6):
  - build the Cartesian product of those opponent move sets
  - keep the highest-scoring joint responses, capped at 6 combinations

opponent_aware_root_scores(state, snake_id):
  For each of our legal root moves:
    1. Simulate that move against each joint opponent response
    2. Evaluate the resulting state with mcts_evaluate_state()
    3. Aggregate scenario values as:

       0.55 * worst
       + 0.30 * average
       + 0.15 * median

This is not full minimax search. It is a one-step adversarial robustness
estimate used only to shape the root priors before the tree search begins.

------------------------------------------------------------------------------
§6.6  Rollout
------------------------------------------------------------------------------

rollout(state, snake_id, config) iterates up to rollout_depth steps.

At each step:
  - If terminal, stop.
  - If heuristic rollouts are enabled, choose rollout_move().
  - Otherwise choose a random legal move.
  - Apply advance_state(..., stochastic=True).

The function returns:
  - reward from mcts_evaluate_state()
  - a list of our rollout moves

That move list is only needed by the RAVE path.

------------------------------------------------------------------------------
§6.7  Backpropagation
------------------------------------------------------------------------------

backpropagate(node, reward, moves_in_rollout, use_rave):
  - increments visits
  - adds reward to value
  - adds reward^2 to value_sq
  - if RAVE is enabled, updates AMAF tables for all rollout moves

The same scalar reward is propagated unchanged to all ancestors. There is no
player alternation or sign inversion because the tree is always from the
perspective of the controlled snake only.

------------------------------------------------------------------------------
§6.8  Final move choice and safety override
------------------------------------------------------------------------------

After the search budget expires:
  - The chosen root child is the one with highest average reward
    and visit count as a tiebreak.

agent_server.choose_move() then applies two checks:

  1. Illegal-choice recovery
     If the chosen MCTS move is not legal in the current real state,
     replace it with the heuristic move.

  2. Heuristic safety override
     If:
       - MCTS chose a different move from the heuristic
       - the chosen child average exists
       - that average is negative
       - the heuristic's score exceeds the chosen move's heuristic score by > 60
     then the heuristic move overrides the MCTS move.

This is a practical safeguard against low-sample or pathological root choices.

================================================================================
§7  AGENT SERVER FLOW
================================================================================

The request flow for a live move is:

  1. main.py reads SNAKE_TACTIC and calls choose_move(game_state, mode).
  2. choose_move() converts API JSON into internal state.
  3. It also computes heuristic_choice and heuristic_scores immediately.
  4. If mode is:
     - random    -> random legal move
     - heuristic -> heuristic_choice
     - mcts*     -> run mcts_move()
  5. MCTS output is checked against legality and the heuristic safety override.
  6. The final move is returned as {"move": move_name}.

The agent server also assigns cosmetic metadata in info(mode):
  each variant has a separate colour, head, and tail preset.

================================================================================
§8  HEADLESS SIMULATOR AND TOURNAMENT TOOLING
================================================================================

simulate_game.py provides a local engine for evaluation without the Battlesnake
server. It uses the same internal state model and the same advance_state()
transition logic, which keeps local experiments aligned with the agent logic.

Important simulator properties:

  1. Initial spawns
     Snakes start from standard corner positions on 11x11 boards.

  2. Food spawning
     Minimum 2 food items are maintained.
     Additional food spawns with probability 0.25 each turn.

  3. Hazard pits
     For 11x11 boards, fixed pit centres are:
       (3,3), (3,7), (7,3), (7,7)

     Hazard stacks follow:
       - turn < 26: no hazards
       - turns 26-175: ramp to a max of 4 stacks
       - turns 176+: drain every 25 turns

  4. API bridge
     _internal_to_api() reconstructs a Battlesnake-like game_state dict so
     local agent callables can be evaluated through the same interface as the
     live server path.

tournament.py layers on top of simulate_game.py.
It is designed for reproducible headless comparisons:
  - pairwise round-robin scheduling
  - explicit seed control
  - optional multiprocessing
  - optional TrueSkill if installed
  - CSV export of game-level records and ratings

This tooling exists in the codebase, but this document does not interpret or
report any of its outputs.

================================================================================
§9  KEY DESIGN CHOICES REFLECTED IN THE CURRENT CODE
================================================================================

Choice 1: Shared transition logic
  The same advance_state() function is used by heuristic lookahead, MCTS, and
  the headless simulator. This reduces rule drift across modules.

Choice 2: Two evaluators instead of one
  evaluate_state() is richer and large-scale for heuristic move ranking.
  mcts_evaluate_state() is bounded and rollout-friendly for tree search.
  This separation is deliberate and appropriate.

Choice 3: Rich root priors, cheap internal priors
  The code spends computation at the root where it matters most, then keeps
  deeper-node priors cheap to preserve rollout throughput.

Choice 4: Greedy opponent model, not opponent tree search
  Opponents are approximated by heuristic move selection rather than full MCTS.
  This is a computational compromise to keep the search tractable.

Choice 5: Soft stochasticity in search
  Search expansions and rollouts do not assume perfectly deterministic opponent
  play. Instead they randomise among near-best moves, which broadens coverage.

Choice 6: Heuristic guardrail above MCTS
  The final move selection is not purely "trust the tree". The code explicitly
  preserves a heuristic veto when the MCTS estimate is negative and clearly
  inferior by heuristic standards.

================================================================================
§10  KNOWN LIMITATIONS AND MISMATCHES
================================================================================

Limitation 1: teamname.py docstring is stale
  teamname.py still claims the contest agent uses RAVE/AMAF as part of "all
  improvements". In the actual config, ALL_IMPROVEMENTS_CONFIG sets:
    use_rave=False
    use_ucb1_tuned=True

Limitation 2: Single-agent tree perspective
  The tree only branches on our actions. Opponent decisions are approximated
  through greedy or shortlist-stochastic policies rather than represented as
  explicit adversarial tree nodes.

Limitation 3: No tree reuse between turns
  MCTS rebuilds from scratch every move. This is simpler and robust, but it
  discards information gathered on the previous turn.

Limitation 4: Short rollout horizon
  rollout_depth is 8. Long tactical traps or delayed starvation sequences can
  still be missed, especially when they lie beyond that horizon.

Limitation 5: Heuristic bias inside search
  Priors, greedy opponent modelling, and heuristic rollouts all inject domain
  bias into MCTS. This is useful under a time budget, but it means the search
  is not an unbiased Monte Carlo estimator.

Limitation 6: Simulator is aligned but not authoritative
  simulate_game.py approximates the target environment for local studies, but
  the live Battlesnake engine remains the ground truth for official behaviour.
